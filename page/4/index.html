流畅的Python_读书笔记(一)

发表于 2017-06-05 | 分类于 Python_Notes

这周正式开始阅读另一本Python的书籍《流畅的Python》，这周阅读的部分有Python数据模型、序列构成的数组和字典和集合。下面分享我的读书笔记。

首先书中用一个简单的代码展示了如何实现__getitem__和__len__特殊方法。

import collections

Card = collections.namedtuple('Card', ['rank', 'suit'])

class FrenchDeck:
    ranks = [str(n) for n in range(2, 11)] + list('JQKA')
    suits = 'spades diamonds clubs hearts'.split()
    
    def __init__(self):
        self._cards = [Card(rank, suit) for suit in self.suits for rank in self.ranks]
        
    def __len__(self):
        return len(self._cards)
    
    def __getitem__(self, position):
        return self._cards[position]

针对上述代码，我最大的启发就是Python特殊方法真的很有用。通过实现__getitem__和__len__特殊方法，我们直接可以用len()计算卡片的长度，而且这个函数也变成了可迭代的。

from math import hypot

class Vector:
    
    def __init__(self, x=0, y=0):
        self.x = x
        self.y = y
    
    def __repr__(self):
        return 'Vector(%r, %r)' % (self.x, self.y)
    
    def __abs__(self):
        return hypot(self.x, self.y)
    
    def __bool__(self):
        return bool(abs(self))
    
    def __add__(self, other):
        x = self.x + other.x
        y = self.y + other.y
        return Vector(x, y)

    def __mul__(self, scalar):
        return Vector(self.x * scalar, self.y * scalar)

散列表其实是一个稀疏数组（总是有空白元素的数组称为稀疏数组）。散列表里的单元叫做表元（bucket）。在dict的散列表当中，每个键值对都占用一个表元，每个表元都有两个部分，一个是对键的引用，另一个是对值的引用。因为所有表元的大小一致，所以可以通过偏移量来读取某个表元。

而如果要把一个对象放入散列表，首先要计算元素键的散列值。Python中可以通过hash( )方法来做这件事情。

有几个方面需要注意：

  1. 散列值和相等性

    如果两个对象在比较的时候是相等的，那它们的散列值必须相等。为了让散列值能够胜任散列表索引这一角色，它们必须在索引空间中尽量分散开来。因此在最理想的情况下，越是相似但不相等的对象，它们的散列值的差别应该是越大。

  2. 散列表算法

    为了获取mydict[search_key]背后的值 ，Python首先会调用hash(search_key)来计算search_key的散列值，把这个值最低的几位数字当作偏移量，在散列表里查找表元（具体取几位，得看当前散列表的大小）。

  3. 键必须是可散列的

    一个可散列的对象必须满足以下要求：

    1. 支持hash()函数，并且通过__hash__()方法所得到的散列值是不变的。

    2. 支持通过__eq__()方法来检测相等性。

    3. 若 a == b为真，则hash(a) == hash(b) 也为真。

  4. 字典在内存上的开销巨大

    由于字典使用了散列表，而散列表又必须是稀疏的，这导致它在空间上的效率低下。

  7. 往字典里添加新键可能会改变已有键的顺序

讀《計算廣告》筆記(一)

发表于 2017-06-02 | 分类于 讀書筆記

發現對於我個人而言我根本不了解廣告。廣告的學問比我想像中要更多更深，因此我買了一本書《計算廣告》來多瞭解廣告，包括在線廣告市場與背景、在線廣告產品邏輯、以及計算廣告關鍵技術。那麼就以筆記這種方式，來加強我對書中內容的理解吧。

書中首先給出了廣告的定義：

廣告是由已確定的出資人通過各種媒介進行的有關產品（商品、服務和觀點）的，通常是有償的、有組織的、綜合的、勸服性的非人員的信息傳播活動。

首先，書中提到，出資人、媒體和受眾這三者的利益博弈關係是廣告活動永遠的主線，這一主線將貫穿於商業和產品型態的整個演化過程。出資人指的是需求方（demand），一般是廣告主（advertiser）、代表廣告主利益的代理商（agency）或者其他技術型態的採買方；媒體一般是供求方，同時也可以是其他技術型態的變現平台。

書中也給出了廣告的目的：

廣告的根本目的是廣告主通過媒體達到低成本的用戶接觸。

也就是說，按某種市場意圖接觸響應的人群，進而影響其中的潛在用戶，使他們選擇廣告主產品的機率增加，或者對產品性價比的苛求程度降低，這才是廣告的根本目的。

對於互聯網廣告，作者認為：

一切付費的信息、產品或服務的傳播渠道，都是廣告

在線廣告創意有以下七種類型：

  1. 橫幅廣告（banner ad）
  2. 文字鏈廣告（textual ad）
  3. 富媒體廣告（rich media ad）：這類廣告往往是利用視覺衝擊力較強的表現形式，在不佔用固定版面位置的情況下，向用戶侵入式地投送廣告素材。常見形式有彈窗、對聯、全屏等。它比較適合在高質量媒體做一些品牌性質比較強的廣告投放，但是對用戶的使用體驗往往影響比較大。
  4. 視頻廣告（video ad）：隨著在線視頻的快速發展，在視頻流播放的間隙插入的廣告也成為互聯網廣告的一種重要形式。
  5. 社交廣告（social ad）：即在社交網絡環境下嵌入的廣告。這種方式力求在用戶自然關注的交互過程中盡可能自然地插入廣告，希望通過用戶的擴散式傳播獲得更大的影響力和口碑。
  6. 移動廣告（mobile ad）：即在應用中插入廣州的SDK（Software Development Kit）和響應的廣告網絡（背景是移動互聯網在近幾年的爆發式增長）
  7. 郵件定向營銷廣告（E-mail Direct Marketing, EDM）：通過電子郵件的方式向目標用戶傳遞推廣信息的一種網絡營銷手段。

然後書中給了大量專業名詞：

  • 整合營銷（integrated marketing）：通過多種渠道的有機配合來達到投放效果的最優。
  • 投入產出比（Return On Investment，ROI）：某次廣告活動的總產出與總投入的比例。
  • 合約廣告（agreement-based advertising）：即採用合同約定的方式確定某一廣告位在某一時間段為某特定廣告主所獨佔，並且根據雙方的要求，確定廣告創意和投放策略。
  • 定向廣告（targeted advertising）: 這樣的廣告系統已經對計算技術產生了兩個具體的需求：一個是受眾定向（audience targeting），即通過技術手段標定某個用戶的性別、年齡或其他標籤。二是廣告投放（ad serving），即將廣告投送由直接嵌入頁面變為實時響應前端請求，並根據用戶標籤自動決策和返回合適的廣告創意。
  • 擔保式投遞（guaranteed delivery， GD）：媒體向廣告主保證某個投放量，並在此基礎上確定合同的總金額以及投放量未完成情況下的賠償方案。
  • 按千次展示付費（Cost per Mille， CPM）
  • 在線分配（online allocation）：一是如何有效地將流量分配到各個合約互相交叉的人群覆蓋上；二是要在在線的環境下實時地完成每一次展示決策。
  • 競價廣告（auction-based adverising）:供給方只向廣告主保證質即單位流量的成本，但不再以合約的方式給出量的保證，也就說，對每一次展示都基本按照收益最高的原則來決策。
  • 廣告網絡（ad Network，ADN）：它批量地運營媒體的廣告位資源，按照人群或上下文標籤賣給需求方，並用競價的方式決定流量分配。
  • 點擊付費（Cost per Click， CPC）
  • 千次展示收益（Revenue per Mille， RPM）
  • 交易終端（Trading Desk， TD）：面向多個ADN或媒體按人群一站式採買廣告並優化投入產出比的需求方產品
  • 實時競價（Real Time Bidding， RTB）：讓需求方按自己的人群定義來挑選流量。它將拍賣的過程由廣告主預先出價，變成每次展示時實時出價。只要把廣告展示的上下文頁面URL以及訪客的用戶標誌等信息傳給需求方，它就有充分的信息完成定制化的人群選擇和出價。
  • 廣告交易平台（ad Exchange， ADX）：大量聚合各媒體的剩餘流量並採用實時競價方式為他們變現的產品型態。
  • 需求方平台（Demand Side Platform， DSP）：通過實時競價的方式，按照定制化的人群標籤購買廣告。
  • 優選（preferred deals）
  • 私有交易市場（Private Market Place，PMP）
  • 原生廣告（Native AD）：將內容與廣告以某種方式統一決策或排序的廣告產品。

爺爺紀念

发表于 2017-06-02

距離爺爺去世，已經過去了大半年了。我還記得，去年十月中旬我在準備考研的時候，忽然收到爸爸的電話，說爺爺身體可能不行了，要我回去一趟。當時我記得我在寢室弄著我的髮型，在學習怎樣用噴霧給頭髮定型。然後我馬上收拾東西，回自習室拿了一些書本，然後買了機票，就回去了。我還依稀記得，回到家那晚，已經快一點了。所以晚上就來不及去看望爺爺。

到了早上，我就去中心醫院看爺爺。當時看到爺爺已經說不出話了，看到他手臂上都是插著管子，然後還時不時要抽痰，我看著真的很心疼。我就說，我是鋭鴻，爺爺我回來啦。那一幕我真的永遠都忘不了。然後爺爺好像哭了，我看到他眼淚流下來。當時其實最遺憾的就是，沒有在他身體健康的時候多陪他。沒有早點功成名就，沒有讓他看到他的孫兒結婚兒孫滿堂。我假裝堅強，忍著不讓淚水流下來，假裝這一切都只不過人生的輪迴，生老病死，人之常情。

二、open函数

我们都知道，open函数可以打开一个文件。open的返回值是一个文件句柄，从操作系统托付给你的Python程序。一旦处理完文件，会想归还这个文件句柄，只有这样程序不会超出一次能打开文件句柄的数量上限。

with open('photo.jpg', 'r+') as f:

open的第一个参数是文件名。第二个(mode打开模式)决定了这个文件如何被打开。

  • 读取文件，传入r
  • 读取并写入文件，传入r+
  • 覆盖写入文件，传入w
  • 在文件末尾附加内容，传入a

我们可以利用io.open指定编码，也就是说能够传入encoding关键字参数来传入编码。利用encoding关键字参数传入编码，能够以正确的指定方式及编码方式打开文件。

三、协程

Python中的协程和生成器很相似但又稍有不同。主要区别在于：

  • 生成器是数据的生产者
  • 协程是数据的消费者

首先回顾一下生成器的创建过程。

>>> def fib():
...    a, b = 0, 1
...    while True:
...        yield a
...        a, b = b, a+b
...        
>>> for i in fib():
...    print i
...

这样做不仅快而且不会给内存带来压力，因为所需要的值都是动态生成的而不是将它们都存储在一个列表中。更概括地说如果现在我们在上面的例子中使用yield便可获得一个协程。协程会消费掉发给它的值。我们来看Python的grep例子。

>>> def grep(pattern):
...    print("Searching for", pattern)
...    while True:
...        line = (yield)
...        if pattern in line:
...            print(line)
...
>>> s = grep('python')
>>> next(s)
('Searching for', 'python')
>>> s.send('python good')
python good

发送的值会被yield接收。运行next()方法是为了启动一个协程。因为协程中包含的生成器不是立即执行，而是通过next()方法来响应send()方法。因此必须通过next()方法来执行yield表达式。然后可以通过close()方法来关闭一个协程。

四、函数缓存

函数缓存允许我们将一个函数对于给定参数的返回值缓存起来。

当一个I／O密集的函数被频繁使用相同的参数调用时，函数缓存可以节约时间。

>>> from functools import wraps
>>> def memoize(function):
...    memo = {}
...    @wraps(function)
...    def wrapper(*args):
...        if args in memo:
...            return memo[args]
...        else:
...            rv = function(*args)
...            memo[args] = rv
...            return rv
...    return wrapper
...
>>> @memoize
... def fibonacci(n):
...    if n < 2: return n
...    return fibonacci(n-1) + fibonacci(n-2)
...
>>> fibonacci(25)
75025

五、上下文管理

上下文管理是允许在有需要的时候，精确地分配和释放资源。对它使用最广泛的案例就是with语句了。

想象当有两个相关操作，你想让它们结对执行，然后在它们中间放置一段代码。

with open('some_file', 'w') as opened_file:
    opened_file.write('Hola!')

上面这段代码会打开一个文件，往里面写入一些数据，然后关闭它。如果在往文件写数据时发生异常，它会尝试去关闭文件，这与下一段代码是等价的。

file = open('some_file', 'w')
try:
    file.write('Hola!')
finally:
    file.close()

当与第一个例子相比时，我们可以看到，通过使用with，许多boilerplate code被消掉了。这就是with语句的主要优势，它确保我们的文件会被关闭，而不用关注嵌套代码如何退出。下面将探讨一下上下文管理的实现方式。

基于类的实现

一个上下文管理器的类，最起码要定义__enter__和__exit__方法。

class File(object):
    def __init__(self, file_name, method):
        self.file_obj = open(file_name, method)
    
    def __enter__(self):
        return self.file_obj
    
    def __exit__(self, type, value, traceback):
        self.file_obj.close()

通过定义__enter__和__exit__方法，我们可以在with语句里使用它。譬如以下代码

with File('some_file', 'w') as opened_file:
    opened_file.write('Hola!')

__exit__函数接受三个参数，这些参数对于每个上下文管理器类的__exit__方法都是必须的。从底层来看发生的事情是这样的：

  1. with语句先暂存了File类的__exit__方法
  2. 然后它调用File类的__enter__方法
  3. __enter__方法打开文件并返回给with语句
  4. 打开的文件句柄被传递给opened_file参数
  5. 使用.write()来写文件
  6. with语句调用之前暂存的__exit__方法
  7. __exit__方法关闭了文件

处理异常

在第4步和第6步之间，如果发生异常，Python会将异常的type、value和traceback传递到__exit__方法中。__exit__方法来决定如何关闭文件以及是否需要其他步骤。在上述例子中，当异常发生时，with语句会采取哪些步骤。

  1. 它把异常的type、value和traceback传递给__exit__方法
  2. 它让__exit__方法来处理异常
  3. 如果__exit__返回的是True，那么这个异常就被优雅地处理了
  4. 如果__exit__返回的是True以外的任何东西，那么这个异常将被with语句抛出。

在案例中，__exit__方法返回的是None(如果没有return语句那么方法会返回None)。因此，with语句抛出了那个异常。

Traceback (most recent call last):
    File "<stdin>", line2, in <module>
AttributeError: 'file' object has no attribute `undefined_function`

尝试下在__exit__方法中处理异常：

class File(object):
    def __init__(self, file_name, method):
        self.file_obj = open(file_name, method)
    
    def __enter__(self):
        return self.file_obj
    
    def __exit__(self, type, value, traceback):
        print("Exception has been handled")
        self.file_obj.close()
        return True

基于生成器的实现

还可以基于装饰器和生成器来实现上下文管理器。

Python有个contextlib模块专门用于这个目的。我们可以使用一个生成器函数来实现一个上下文管理器，而不是使用一个类。

from contextlib import contextmanager

@contextmanager
def open_file(name):
    f = open(name, 'w')
    yield f
    f.close()
  1. Python解析器遇到了yield关键字。因为这个关键字所以创建的是一个生成器而不是一个普通的函数。
  2. 因为这个装饰器，contextmanager会被调用并传入函数名(open_file)作为参数。
  3. contextmanager函数返回一个以GeneratorContextManager对象封装过的生成器
  4. 这个GeneratorContextManager被赋值给open_file函数，实际上是在调用GeneratorContextManager对象。

因此我们可以用这个新建的上下文管理器。

with open_file('some_file') as f:
    f.write('hola!')

三、推导式

推导式（又称解析式）是Python的一种独有特性，可以从一个数据序列构建另一个新的数据序列的结构体。共有三种推导。

  • 列表(list)推导式
  • 字典(dict)推导式
  • 集合(set)推导式

列表推导式(list comprehensions)

列表推导式提供了一种简明扼要的方法来创建列表。

它的结构是在一个中括号里包含一个表达式，然后是一个for语句，然后是0个或多个for或者if语句。返回结果将是一个新的列表。

>>> multiples = [i for i in range(30) if i % 3 is 0]
>>> print(multiples)
[0, 3, 6, 9, 12, 15, 18, 21, 24, 27]

字典推导式(dict comprehensions)

举一个例子，即利用字典推导式快速对换一个字典的键和值。

>>> {v: k for k, v in mcase.items()}
{3: 'Z', 7: 'A', 10: 'a', 34: 'b'}

集合推导式(set comprehensions)

它们跟列表推导也是类似的。

>>> squared = {x **2 for x in [1,2,3]}
>>> print(squared)
set([1, 4, 9])

有三种方法来处理多个异常。

第一种方法需要把所有可能发生的异常放到一个元组里。

try:
    file = open('test.txt', 'rb')
except (IOError, EOFError) as e:
    print("An errror occurred. {}".format(e.args[-1]))

另外一种方式是对每个单独的except语句块中处理。

try:
    file = open('test.txt', 'rb')
except EOFError as e:
    print("An EOF error occurred.")
    raise e
except IOError as e:
    print("An error occurred.")
    raise e

最后一种方法是会捕获所有异常：

try:
    file = open('test.txt', 'rb')
except Exception:
    raise

finally

一般而言是把主程序代码包裹进try从句。然后把一些代码包裹进一个except从句，它会在try从句中的代码触发异常时执行。

然后还有finally从句，包裹到finally从句中的代码不管异常是否触发都将会被执行。这可以被用来在脚本执行之后做清理工作。

try:
    file = open('test.txt','rb')
except IOError as e:
    print('An IOError occurred. {}'.format(e.args[-1]))
finally:
    print("This would be printed whether or not an exception occured!")

...    def __init__(self):
...        self.prev = 0
...        self.curr = 1
...

...        return self
...

...        value = self.curr
...        self.curr += self.prev
...        self.prev = value
...        return value
...

Fib既是一个可迭代对象，因为它实现了__iter__方法。同时又是一个迭代器，因为它实现了next犯法。每次调用next()方法时候做两件事：

  1. 为下一次调用next()方法修改状态
  2. 为这次调用生成返回结果

迭代器就像一个懒加载的工厂，等到有人需要的时候才给它生成值返回，没调用的时候就处于休眠状态等待下一次调用。

3.5 生成器(Generators)

生成器是一种特殊的迭代器，因此它拥有迭代器的所有特性。生成器没有把所有的值存在内存中，而是在运行时生成值，因此其不会一下子占用大量的内存和CPU资源。而且生成器相比于迭代器更加优雅，不需要像上面一样写__iter__()和next()方法，只需要一个yield关键字。

>>> def fib():
...    prev, curr = 0, 1
...    while True:
...        yield curr
...        prev, curr = curr, curr + prev     
...
>>> f = fib()
>>> list(islice(f, 0, 10))
